# Test-Driven Development (TDD)

## Overview
테스트를 먼저 작성합니다. 실패하는 것을 확인합니다. 통과할 만큼의 최소한의 코드만 작성합니다.

**핵심 원칙:** 테스트가 실패하는 것을 직접 보지 않았다면, 그 테스트가 올바른 것을 검증하는지 알 수 없습니다.

## The Iron Law
```
실패하는 테스트 없이는 프로덕션 코드를 작성하지 않는다.
```

테스트보다 코드를 먼저 작성했나요? 삭제하고 처음부터 다시 시작하세요.

## Red-Green-Refactor Cycle

### 1. RED - 실패하는 테스트 작성
무엇이 일어나야 하는지 보여주는 최소한의 테스트를 작성합니다.
- 하나의 동작만 테스트
- 명확한 이름
- 실제 코드 사용 (가급적 모의 객체 지양)

### 2. Verify RED - 실패 확인 (필수)
`npm test` 등을 실행하여 테스트가 **기대하는 이유로 실패**하는지 확인합니다. (오타나 문법 에러가 아닌 기능 미구현으로 인한 실패여야 함)

### 3. GREEN - 최소한의 코드 작성
테스트를 통과하기 위한 **가장 단순한 코드**만 작성합니다. 오버엔지니어링하지 마세요.

### 4. Verify GREEN - 통과 확인 (필수)
테스트가 통과하고, 기존의 다른 테스트들도 여전히 통과하는지 확인합니다.

### 5. REFACTOR - 정리
테스트가 초록색인 상태에서만 진행합니다:
- 중복 제거
- 이름 개선
- 도우미 함수 추출
동작은 변경하지 않고 구조만 개선합니다.

## 지침 및 주의사항
- "이번 한 번만 건너뛰자"는 생각은 금물입니다.
- 테스트가 통과했다면 즉시 다음 기능을 위한 실패하는 테스트로 넘어갑니다.
- 버그 발견 시, 해당 버그를 재현하는 실패하는 테스트를 먼저 작성합니다.
